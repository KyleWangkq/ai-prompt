### 按业务层次分类
- **战略层术语**: 支付模块、支付上下文、限界上下文边界相关术语
- **战术层术语**: 支付单聚合、支付流水实体、支付金额值对象相关术语  
- **技术层术语**: 支付渠道集成、回调处理、Repository接口相关术语

### 按架构层次分类
- **领域层术语**: 支付单、支付状态、支付规则、支付事件等核心业务概念
- **应用层术语**: 支付服务、合并支付、支付用例、DTO等应用协调概念
- **基础设施层术语**: 支付渠道、交易号、回调、数据映射等技术实现概念

### 按重要程度分类
- **核心术语**: 支付单、支付状态、支付金额等架构设计的关键概念，必须精确定义
- **重要术语**: 支付类型、支付渠道、合并支付等影响设计质量的概念，需要清晰定义
- **辅助术语**: 业务标签、优先级等支撑性概念，保持一致性即可影响 |

## 术语使用指南

### 标准化命名规则
1. **业务优先**: 优先使用业务语言，避免纯技术术语
2. **中英对照**: 重要术语提供标准英文对照
3. **简洁明确**: 避免冗长表达，保持概念清晰
4. **一致性**: 在所有文档中保持术语使用的一致性

### 术语扩展原则
1. **业务驱动**: 新术语必须来自真实业务需求
2. **专家确认**: 新术语需要业务专家确认和认可
3. **影响评估**: 评估新术语对现有文档体系的影响
4. **版本管理**: 在Glossary-History.md中记录所有变更

## 术语使用规范 (Term Usage Guidelines)

### 命名约定
- **类名**: 使用英文术语的PascalCase格式，如`Payment`、`PaymentTransaction`
- **方法名**: 使用动词+名词的camelCase格式，如`createPayment`、`updatePaymentStatus`
- **字段名**: 使用camelCase格式，如`paymentId`、`paidAmount`
- **数据库字段**: 使用snake_case格式，如`payment_id`、`paid_amount`

### 术语一致性原则
- **单一性**: 一个概念只使用一个标准术语，避免同义词混用
- **精确性**: 术语使用必须准确反映业务含义，避免技术术语和业务术语混淆
- **稳定性**: 核心术语一旦确定应保持稳定，变更需要经过严格的影响评估

### 跨层使用约定
- **领域层**: 严格使用业务术语，禁止技术实现术语
- **应用层**: 主要使用业务术语，技术术语仅限于协调和编排
- **基础设施层**: 可以使用技术术语，但与业务相关的概念必须保持术语一致性
